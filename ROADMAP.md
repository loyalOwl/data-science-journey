# Дорожная карта AI Engineer / Agent Engineer на 8 недель

Период: **16 июля — 9 сентября 2026 года**.

## Главная цель

Научиться проектировать, создавать, оценивать, развёртывать и сопровождать автономных AI-агентов для разных бизнес-задач. Не стать специалистом по одному фреймворку и не ограничиться одним Customer Success Agent, а понять полный жизненный цикл agentic-системы.

[SoloGPT](https://sologpt.ru/main/) и [«Колли»](https://cally.ru/) используются как компании-полигоны: мы будем искать в их продуктах разные задачи для агентов — продажи, поддержку, контроль качества, работу с базой знаний, анализ диалогов, улучшение промптов, CRM-операции и внутренние процессы.

Подробная постановка профессионального профиля находится в [docs/AI_ENGINEER_PROFILE.md](docs/AI_ENGINEER_PROFILE.md), а разбор компаний-полигонов — в [docs/PRODUCT_REFERENCE.md](docs/PRODUCT_REFERENCE.md).

Перед практической частью можно прочитать [теоретический путеводитель по инженерии AI-агентов](output/pdf/AI_AGENT_ENGINEERING_FIELD_GUIDE.pdf). Он объясняет простым языком общую архитектуру, основные термины и связь всех этапов этого roadmap.

## Что уже стало фундаментом

- настроены Python, виртуальное окружение, Git и GitHub;
- написаны функции, валидация и автоматические тесты;
- освоены основные типы, коллекции, исключения и CSV;
- сформирован цикл `изменение → проверка → commit → push`.

Data Science не отменяется, но сложное ML и углублённая статистика пока не являются главным маршрутом. SQL и анализ данных используются для состояния агентов, evals, продуктовых метрик и отладки.

## Целевая архитектурная картина

```text
задача пользователя, событие или расписание
                    ↓
              API / worker
                    ↓
┌──────────────── Agent Harness ────────────────┐
│ инструкции │ контекст │ состояние │ план      │
│ tools      │ agent loop │ checkpoints         │
│ permissions │ guardrails │ stopping conditions│
└────────────────────┬──────────────────────────┘
                     ↓
       API, MCP, базы данных, файлы, браузер
                     ↓
          результат или действие в системе
                     ↓
      traces + логи + метрики + обратная связь
                     ↓
        offline/online evals и новая версия
```

**Agent harness** — среда вокруг модели, которая даёт агенту инструменты и контекст, ограничивает его действия, сохраняет прогресс и делает качество измеримым.

Переносимый принцип: **модель рассуждает, но инженерная система определяет, что ей доступно, как проверить результат и когда остановиться**.

## Сквозной проект: Agent Harness

Вместо одного приложения все восемь недель развивается переиспользуемый harness:

- единые схемы задач и результатов;
- регистрация инструментов;
- управление контекстом и состоянием;
- лимиты шагов, времени и стоимости;
- retries, timeouts и обработка ошибок;
- human-in-the-loop;
- traces и логи;
- eval-наборы и сравнение версий;
- конфигурация разных агентов поверх общей основы.

Каждый новый агент проверяет, действительно ли harness можно переиспользовать, а не копировать заново.

## Неделя 1. Production-Python и детерминированный workflow — 16–22 июля

Зачем: агентные фреймворки не заменяют инженерную базу. Сначала нужно научиться надёжно представлять данные и выполнять бизнес-логику без LLM.

Темы:

- словари, модули, классы и `dataclass`;
- typing и строгие контракты данных;
- JSON, файлы и конфигурация;
- исключения, логирование и тесты;
- чистые функции и разделение ответственности;
- базовые понятия состояния и идемпотентности.

Проект 1: **Customer Success Core без LLM**. Программа анализирует несколько фактов о клиенте и по прозрачным правилам формирует задачу менеджеру.

Это только первый полигон, а не специализация всего курса.

Готово, когда:

- данные имеют понятную схему;
- ошибки обрабатываются без падения;
- правила отделены от ввода и вывода;
- тесты покрывают обычные и пограничные сценарии;
- Кирилл может объяснить путь данных через программу.

## Неделя 2. LLM-примитивы и собственный agent loop — 23–29 июля

Зачем: понять работу агента до подключения фреймворка.

Темы:

- OpenAI API и provider-neutral интерфейс модели;
- инструкции, сообщения, токены и контекстное окно;
- structured outputs;
- function calling и строгие схемы tools;
- цикл `модель → tool → наблюдение → следующий шаг`;
- stopping conditions, лимиты шагов и стоимости;
- mocks и тесты без платных запросов;
- первые eval-примеры.

Проект 2: **Support Triage Agent**. Агент классифицирует обращение, запрашивает недостающие данные, использует один безопасный инструмент и передаёт сложный случай человеку.

Готово, когда Кирилл может вручную написать и объяснить простой agent loop без Agents SDK, LangChain или LangGraph.

## Неделя 3. AI Backend: FastAPI, SQL и выполнение задач — 30 июля–5 августа

Зачем: превратить agent loop в сервис, способный принимать задачи, хранить состояние и взаимодействовать с бизнес-системами.

Темы:

- HTTP, REST, FastAPI и Pydantic;
- async/await;
- SQL и SQLite/PostgreSQL;
- задачи, запуски, статусы и история шагов;
- webhook, retries, timeouts и idempotency;
- фоновые worker-задачи;
- тесты API и безопасная конфигурация.

Артефакт: **Agent Service v1** — API для запуска двух разных агентов на общем runtime с сохранением статуса и результата.

Готово, когда повторный запрос не создаёт опасных дублей, а прерванный запуск имеет понятный статус.

## Неделя 4. RAG и Context Engineering — 6–12 августа

Зачем: научить агента выбирать правильный контекст, а не помещать в prompt все доступные данные.

Темы:

- embeddings, chunks, metadata и vector search;
- ingestion и обновление документов;
- retrieval, reranking и цитирование источников;
- краткосрочная и долгосрочная память;
- context budgeting и compaction;
- защита от prompt injection;
- evals retrieval и groundedness.

Проект 3: **Knowledge and Document Agent**. Он ищет информацию в документах, выполняет несколько шагов исследования, показывает источники и отказывается от неподтверждённых выводов.

Готово, когда отдельно измеряются качество поиска, качество ответа и корректность отказа.

## Неделя 5. Архитектуры и фреймворки — 13–19 августа

Зачем: увидеть одинаковые архитектурные элементы в разных реализациях и научиться выбирать инструмент под задачу.

Практическое сравнение одной задачи:

| Подход | Что изучаем |
|---|---|
| чистый Python + Responses API | как устроен agent loop без скрытой магии |
| OpenAI Agents SDK | tools, guardrails, handoffs, sessions и tracing |
| LangChain | готовые abstractions, integrations и middleware |
| LangGraph | state graph, checkpoints, durable execution и human-in-the-loop |

Дополнительно:

- workflow против автономного агента;
- router, orchestrator-workers, evaluator-optimizer;
- один агент против multi-agent;
- planning, reflection и self-critique;
- критерии выбора архитектуры.

Артефакт: **Framework Lab** — одна задача реализована минимум двумя способами, а в README зафиксированы компромиссы.

Готово, когда Кирилл способен выбрать подход по требованиям, а не по популярности библиотеки.

## Неделя 6. MCP, долгие задачи и multi-agent — 20–26 августа

Зачем: подключать внешние возможности стандартно и надёжно выполнять задачи, которые не помещаются в один короткий запуск.

Темы:

- MCP client/server, tools, resources и prompts;
- permissions и подтверждение изменяющих действий;
- state, checkpoints, pause/resume и recovery;
- артефакты передачи работы между запусками;
- subagents и параллельная работа;
- объединение результатов и разрешение конфликтов;
- sandbox для выполнения потенциально опасных действий.

Проект 4: **Research Operations Agent**. В простой версии один агент планирует и выполняет исследование; затем мы проверяем, даёт ли разделение на subagents измеримое улучшение.

Артефакт harness: возможность остановить задачу, продолжить её и восстановить ход выполнения.

## Неделя 7. Evals, Observability и self-improving loops — 27 августа–2 сентября

Зачем: production-агента нельзя сопровождать по ощущениям.

Темы:

- traces, spans, логи и продуктовые метрики;
- offline и online evals;
- проверка финального ответа, tool calls и всей trajectory;
- детерминированные graders, human review и LLM-as-judge;
- regression-наборы и сравнение версий;
- A/B и shadow testing;
- reflection и iterative refinement;
- сбор обратной связи;
- предложение изменений промпта или skill;
- sandbox-проверка и обязательное одобрение релиза человеком.

Проект 5: **Agent QA and Improvement Agent**. Он анализирует неудачные traces другого агента, группирует ошибки и предлагает улучшение. Самостоятельно применять изменение в production ему запрещено.

Готово, когда новая версия считается лучше только после измеримого сравнения и проверки регрессий.

## Неделя 8. Production, деплой и портфолио — 3–9 сентября

Зачем: агент ценен, только если его можно воспроизводимо запустить, наблюдать, ограничивать и обновлять.

Темы:

- Docker и конфигурация окружений;
- CI и автоматические проверки;
- деплой API и worker;
- секреты и минимальные права;
- очереди и базовые принципы масштабирования;
- стоимость, latency и rate limits;
- безопасность данных и audit trail;
- runbook: как диагностировать и восстанавливать систему.

Итоговое портфолио:

1. **Customer Success Core** — детерминированная бизнес-логика.
2. **Support или Knowledge Agent** — tools, RAG и handoff.
3. **Research Operations Agent** — длинное выполнение, state и MCP.
4. **Agent QA and Improvement Agent** — evals, traces и контролируемый feedback loop.
5. **Reusable Agent Harness** — общая инфраструктура, на которой запускаются разные агенты.

Не все проекты обязаны быть одинаково большими. Один становится главным production-capstone, остальные доказывают широту архитектурного мышления.

## Уровни автономности, которые нужно пройти

```text
LLM-вызов
   ↓
workflow с фиксированными шагами
   ↓
agent loop с выбором tools
   ↓
stateful long-running agent
   ↓
multi-agent при доказанной пользе
   ↓
контролируемый self-improving loop
```

Чем выше автономность, тем обязательнее sandbox, permissions, stopping conditions, evals, traces и human-in-the-loop.

## Еженедельный кругозор

Раз в неделю выделяем короткий обзор без смены основного практического стека:

- доступные семейства LLM и их различия;
- agent frameworks и runtimes;
- vector databases и способы retrieval;
- observability/evaluation платформы;
- облачные и self-hosted варианты;
- новые production-подходы и их зрелость.

Цель обзора — понимать карту индустрии, а не устанавливать каждый инструмент.

## Минимум за 6 недель

Если срок сжимается:

1. недели 1–4 пройти полностью;
2. на неделе 5 глубоко сравнить чистый loop и один SDK, остальные изучить обзорно;
3. объединить MCP, checkpoints, evals и tracing в один небольшой долгий агент;
4. обязательно выполнить Docker, деплой, README и демонстрацию;
5. multi-agent и автоматическое предложение улучшений оставить после выхода на работу, если они не нужны в первых рабочих задачах.

## Критерии готовности к работе

Кирилл умеет:

- объяснить отличие workflow от автономного агента;
- построить agent loop и tool calling без фреймворка;
- выбрать уровень автономности по риску и неопределённости задачи;
- создать FastAPI/SQL backend для запусков агента;
- реализовать RAG с отдельной оценкой retrieval и ответа;
- подключить инструменты напрямую и через MCP;
- сохранять state, останавливать и продолжать длинную задачу;
- объяснить, когда нужен LangGraph, LangChain или Agents SDK;
- читать trace и воспроизводить ошибку;
- создавать eval-набор и находить регрессии;
- безопасно организовать human-in-the-loop;
- оценить latency, стоимость и потенциальный ущерб;
- развернуть агента через Docker и объяснить его архитектуру за 5–7 минут.
