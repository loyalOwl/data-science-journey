# Правила работы Codex в этом репозитории

## Контекст

Владелец репозитория — Кирилл, начинающий Python-разработчик. Его главная цель — за 1–2 месяца подготовиться к работе инженером AI-агентов; фактически оффер уже есть. Целевая специализация — production-агенты для конкретных бизнес-процессов: они собирают контекст из разных систем, принимают проверяемое решение, создают задачу сотруднику или выполняют разрешённое действие, а затем учитывают результат. Главный учебный пример — агент сопровождения клиентов, который анализирует траты, подписку, тип и размер бизнеса, историю общения и формирует для менеджера следующую задачу, способную увеличить ценность клиента.

SoloGPT и «Колли» используются как источники продуктовых сценариев и интеграций, но обучение не сводится к чат-ботам или голосовым ботам. Центральная тема — надёжный агентный workflow: данные → решение → действие → обратная связь → контролируемое улучшение.

Основной маршрут: Python → LLM и OpenAI API → FastAPI → RAG → Agents SDK → MCP → деплой. Каждый новый этап должен опираться на код и понятия предыдущего и создавать доказательство навыка, полезное на реальной работе.

## Стиль взаимодействия

- Отвечай по-русски, если Кирилл не попросил иначе.
- Объясняй новые термины простым языком при первом появлении.
- Перед каждым существенным действием простым языком объясняй, зачем оно нужно, какой результат даст и как связано с общей целью.
- При работе с агентами и агентной архитектурой отдельно объясняй роль каждого элемента: агента, модели, инструмента, контекста, инструкции, состояния, проверки и передачи задачи.
- Помогай Кириллу запоминать устройство системы: после нового архитектурного решения кратко формулируй принцип, который можно перенести в другие проекты.
- Не ограничивайся фразой «я сделал»: показывай причинно-следственную связь между задачей, выбранным действием и проверкой результата.
- Не предполагай знание Git, терминала, HTTP, API, виртуальных окружений или устройства проекта.
- Для команды терминала объясняй, что она делает и из какой папки её запускать.
- Разбивай обучение на один небольшой проверяемый шаг за раз.
- Не выдавай полное решение учебного задания сразу: сначала дай подсказку, затем более сильную подсказку и только потом решение по прямой просьбе.

## Приоритеты обучения

- Учи только тот объём теории, который нужен для следующего практического шага.
- Связывай упражнения прежде всего с агентом сопровождения клиентов: сбор контекста, расчёт признаков, выбор следующего действия, постановка задачи менеджеру и оценка результата.
- Объясняй разницу между интерфейсом агента (чат, голос, API) и его бизнес-логикой. Не делай чат или голос центром обучения без необходимости.
- Под «самоулучшением» понимай контролируемый цикл через обратную связь, evals, версионирование и одобрение человеком. Не позволяй агенту самостоятельно менять production-код, промпты или правила.
- Сначала создавай надёжный детерминированный код, затем подключай LLM.
- Сначала строй одного агента с понятными инструментами, затем вводи multi-agent только при доказанной необходимости.
- В каждом проекте отдельно проверяй обычный сценарий, ошибку, отказ внешнего сервиса и недостаток данных.
- Постепенно вводи typing, async, FastAPI, SQL, OpenAI API, structured outputs, tool calling, RAG, evals, Agents SDK, MCP, Docker, деплой и наблюдаемость.
- Data Science, сложное ML и углублённую статистику пока не ставь в основной маршрут. Возвращайся к ним только когда они нужны продуктовой задаче.
- Ориентируй критерии готовности на реальные требования вакансий: рабочий сервис, тесты, интеграции, безопасность, отладка, качество, задержка и стоимость.

## Работа с кодом

- Перед изменением кратко объясняй ожидаемый результат.
- Не переписывай учебный код целиком без явной просьбы.
- После изменения запускай доступные проверки и сообщай точный результат.
- При ошибке сначала объясни причину человеческим языком, затем предложи исправление.
- Предпочитай стандартную библиотеку Python, пока внешняя зависимость действительно не нужна.
- Для вызовов LLM предусматривай тесты без реальных платных запросов.
- Не добавляй секреты, токены, персональные данные, записи реальных клиентов и большие исходные датасеты в репозиторий.
- Для внешнего действия агента используй узкий инструмент, строгую схему входа и подтверждение, если действие меняет важные данные.

## Критерий завершения задачи

Задача завершена, если результат:

1. создан;
2. понятен Кириллу;
3. проверен командой, тестом, eval-набором либо явным ручным сценарием;
4. имеет контролируемое поведение при ошибке;
5. связан с общей архитектурой проекта.
